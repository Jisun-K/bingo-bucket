# 🎯 Bingle — 나의 계획이 완성 되는 순간

> **Bingle**은 단순히 해야 할 일을 적는 게 아니라, <br>
>  '하고 싶은 일'을 하나씩 채워가는 감성 빙고 서비스 입니다.<br>
> _"작은 일이라도 괜찮아요 - 중요한 건 '채워가는 과정'이니까."_🌿

---

## 🧭 Overview

**Bingle**은 _Bingo_ + _Mingle_의 합성어예요.  
‘빙고처럼 목표를 완성해 나가는 재미’와 ‘사람들과 섞이고 공감하는 경험’을 담고 싶었어요.  

처음엔 개인용으로 시작했지만,  
추후엔 **다른 사람들과의 계획을 공유하고 공감하는 플랫폼**까지 아우르는 서비스를 목표로 하고 있어요.

> 단순한 ‘투두리스트’가 아닌,  
> 완성의 즐거움을 ‘빙고판’으로 표현하는 것이 핵심입니다. 🪄

## ✨ Key Features

✅ **직관적인 빙고 UX**
- 클릭 한 번으로 목표 추가/완료  
- 완성 시 스타일 변화로 성취감 표현 
- getBingoLines / checkBingo로 완성 라인 계산  

🎨 **테마별 색상 시스템**
- 글로벌 변수 기반 커스텀 테마 지원

🧠 **Zustand 상태 관리**
- 모달, 빙고판 상태를 간결하고 효율적으로 관리  

📱 **반응형 레이아웃**
- 모바일·데스크탑 모두 깔끔하게 표현되는 Grid 구조  

🪶 **확장 가능성 구조**
- 커뮤니티, 공유, 히스토리 기능 등으로 유연하게 확장 가능

## 💡 배운 점

> “처음엔 단순한 기능이었는데, 구조를 잡는 게 진짜 핵심이었어요.”

- **Zustand + Hooks 설계 패턴의 장점**  
  전역 상태를 최소화하면서 모듈 단위로 로직을 나누는 구조를 시도했어요.
  Angular처럼 명확한 구조가 있는 프레임워크에 익숙하다 보니,
  React의 자유로운 패턴이 처음엔 오히려 낯설고 혼란스럽기도 했어요.
  그래도 직접 구조를 짜면서 ‘유연함 속에서도 일관성을 만드는 법’을 조금은 알게 된 것 같아요. 

- **컴포넌트 간 의존도 줄이기**  
  `useBingoBoard`, `useModalStore` 등으로 흐름을 명확히 분리해  
  복잡도를 줄일 수 있던 거 같아요.  

- **‘디자인 감각’이 코드에 영향을 미친다는 점**  
  단순히 잘 동작하는 것보다, 감정이 담긴 UI가 주는 힘이 크다는 걸 느꼈어요.

## 🧩 Troubleshooting

| 문제 | 원인 | 해결 |
|------|------|------|
| `useEffect` 중복 호출 | React Strict Mode의 이중 렌더링 | 개발 모드에서만 발생, 구조 문제 아님 |
| `disabled` 상태가 의도치 않게 변경됨 | 상태 관리가 한 레벨 위에서 공유됨 | 개별 아이템 단위로 상태 분리 후 해결 |
| 글자 수 제한 미작동 | `value` 업데이트 시점 불일치 | `handleChange`에서 조건문 + 버튼 비활성화 처리로 해결 |
